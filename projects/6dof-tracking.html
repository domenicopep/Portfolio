01 — Context

Why tracking matters

Surgical navigation, robotic guidance, and augmented reality all depend on the same fundamental capability: knowing exactly where an object is in 3D space, in real time. NDI tracking systems are the clinical gold standard for this, but interfacing with them requires handling low-level binary protocols, coordinate system transforms, and high-frequency data streams.

This project built a clean, performant interface layer between NDI hardware and downstream applications — abstracting the protocol complexity while maintaining the precision and throughput the hardware provides.

02 — Protocol Layer

Binary protocol parsing

NDI trackers communicate through a proprietary binary protocol with two frame formats: BX (legacy) and BX2 (extended). Each frame contains packed header bytes, tool handle identifiers, transformation matrices encoded as quaternion + translation pairs, and CRC checksums.

The parser handles frame synchronization, endianness conversion, CRC validation, and graceful recovery from corrupted or dropped frames — critical for maintaining tracking continuity during surgical procedures.

  • BX and BX2 binary frame parsing
  • CRC-16 checksum validation
  • Automatic frame synchronization
  • Graceful corruption recovery
  • Multi-tool concurrent tracking
03 — Coordinate Transforms

Quaternion mathematics

The tracker reports orientation as unit quaternions — a compact, singularity-free rotation representation. The interface provides utilities for quaternion-to-matrix conversion, quaternion interpolation (SLERP) for smoothing, and coordinate system transformations between tracker space, patient space, and display space.

These transforms are the foundation that allows downstream applications — AR overlays, robotic controllers, visualization tools — to operate in their own native coordinate systems while maintaining spatial coherence with the physical world.

04 — Streaming Architecture

High-frequency data modes

For applications requiring continuous position updates — real-time AR, robotic control loops, motion analysis — the interface supports streaming data modes that push frames at the tracker's native rate. The architecture uses an asynchronous event-driven model to decouple frame acquisition from processing, preventing dropped frames during computationally expensive downstream operations.
